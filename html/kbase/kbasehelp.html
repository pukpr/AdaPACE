Introduction

This is a very quick introduction to Prolog (the language of AES), designed to get you up to speed in the basics. This will not explain all of Prolog, so if you are interested in a better, more thorough explanation, try one of these books:

"Programming in Prolog", by Clocksin and Mellish. This is the original de facto language refererence manual, but provides a good introduction to the language.

"The Craft of Prolog", by Richard O'Keefe. This a newer book and is very good for seeing how a real expert goes about using Prolog.

 

Syntax

Variables and atoms Prolog identifiers are the usual letter followed by any number of letters, numbers or underscores. However, case is significant, e.g.

 

mary /= Mary /= marY

 

In particular, variables are identified by an initial capital letter or underscore, so Name and _name are variables, while name is an atom. Alternatively, any quoted identifier is an atom, so "Name" and "this is an atom!" are both atoms. The underscore by itself is a special case; it is the anonymous variable (which binds to anything).

 

Lists and structures A list of items is enclosed in square brackets, as in [a,b,c]. A list can also be separated into its head and tail with the form [a|X]. A structure is simply name(arg1), where there can be any number (> 0) of arguments, separated by commas.

 

Comments Comments in Prolog begin with % and extend to the end of the current line.

 

Clauses: Facts, Rules and Procedures A fact is written as

 

likes(john, mary).

 

where the name of the fact states the relationship and the objects are listed in the parenthesis. This fact would be read as "John likes Mary".

A rule is written:

 

likes(john, mary) :- likes(mary, popcorn), brown_eyes(mary).

 

where the :- is read as "if." This rule would be read as "John likes Mary if Mary likes popcorn and Mary has brown eyes." The part of the rule before the :- is called the head and part after is called the body. The body can consist of any number of goals; goals are connected by a comma (,) to indicate a conjunction, i.e. they must all be true for the clause to be true. A procedure is the set of clauses with the same name and arity (number of arguments). The clauses of a procedure do not have to be contiguous, but it is considered good style to keep them together.

 

Execution

Queries Prolog begins execution in response to a query to from the user. If your program contained only the following fact:

 

likes(john, mary).

 

you could give it the query

 

likes(john, mary)?

 

and it would respond yes . On the other hand, if you gave it the query

 

likes(john, ellen)?

 

it would respond no. A no response from Prolog means that it couldn't prove your query. This is not the same as saying your query is false (John just might like Ellen), only that Prolog cannot prove that the query is true in the context of your program. If your program were changed to read:

 

likes(john, mary) :- likes(mary,popcorn), brown_eyes(mary).

likes(mary, popcorn).

brown_eyes(mary).

 

then, in response to the query

 

likes(john, mary)?

 

Prolog would first try to prove likes(mary, popcorn) and then brown_eyes(mary). Both of these goals would succeed and so Prolog would respond yes. If either of the facts were missing, the query would fail. When there are multiple clauses in a procedure, as in likes above, each clause is tried in order until one succeeds.

 

Using Variables All this is lots of fun, but it isn't too interesting until we introduce variables. The scope of a variable is the clause that it appears in; there are no global variables as in most other languages. A variable initially has no value (it is said to be 'uninstantiated' or 'unbound'). When you use a variable Prolog tries to match ('unify') it with the appropriate clauses. A variable's value can't be changed once it is set, but if a goal can't be proven with a certain value, the code will backtrack and try to find another value that will allow the goal to succeed. For example, given the program above, the query likes(john, X)? , generates the response X = mary, meaning your query is provable and the value of X is mary. (If you want more than one answer, type multiple? , and repeat the query until it responds no when no more solutions exist.)

Variables can also be used in rules, as in this definition of the grandfather relation:

 

grandfather(X, Z) :- father(X, Y), parent(Y, Z).

parent(X, Y) :- father(X, Y).

parent(X,Y) :- mother(X, Y).

father(john, tom).

father(john, mary).

father(tom, sue).

mother(mary, harry).

 

The grandfather relation is read as "X is the grandfather of Z if X is the father of some person Y and that person Y is the parent of Z." The parent relation is read as "X is the parent of Y if X is the father of Y or X is the mother of Y". Given the program above, grandfather(john, X)? responds X = sue , and on backtracking responds X = harry .

 

Tracing When you are debugging your code you'll find the trace goal useful, as in trace? , to scroll through the code.

?

 

Defined as query P.

 

P?

 

A clause ending with a question mark is a query.

 

.

 

The fact or clause terminator is '.'

 

P.

 

If the body of a clause is empty, the clause is called a unit clause or a fact, where P is the head literal. We interpret this declaratively as

P is true

and procedurally as

P can be satisfied.

 

if

 

P if Q and R and S

 

P :- Q, R, S.

 

If the body of a clause is nonempty, the clause is called a nonunit clause or rule, where P is the head and Q, R and S are the goals which make up the body. We can read such a clause either declaratively as

P is true if Q and R and S are true

or procedurally as

To satisfy goal P, satisfy the goals Q, R and S.

 

 

->

 

when P succeeds then do Q

 

P -> Q

 

This is defined internally as

 

P -> Q :- P, !, Q.

 

!

 

Cut (discard) all choice points made since the parent goal started execution.

 

!

 

Besides the sequencing of goals and clauses, Prolog provides the cut construct for specifying control information. It is inserted in the program just like a goal. However, it is not a part of the logic of the program and can be ignored as far as the declarative semantics is concerned.

 

The effect of the cut symbol is as follows. When it is first encountered as a goal, cut succeeds immediately. If backtracking returns to the cut, then the parent goal is failed. In other words, the cut operation commits the system to all choices made since the parent goal was invoked, and causes other alternatives to be discarded. Thus, the selected goals include the parent goal itself, any goals occurring before the cut in the clause containing the cut, and any subgoals that were executed during the execution of those preceding goals.

assert

 

Assert clause C.

 

assert(C)

 

The current instance of C is interpreted as a clause and is added to the program (with new private variables replacing any uninstantiated variables). The new clause is positioned after the last clause asserted. C must be instantiated to a non-variable.

The clause assertz(C) is the same as assert(C), so that the new clause becomes the last clause for the procedure concerned.

 

 

asserta

 

Assert C as first clause.

 

asserta(C)

 

This clause is similar to assert(C), except that the new clause becomes the first clause for the procedure concerned.

call

 

Execute the goal X

 

call(X) or X

 

If X is instantiated to an acceptable term in body of a clause, the goal X (i.e. call(X) where call() need not be typed) is executed exactly as if that term appeared textually, except that any cut (!) occurring in X will remove only those choice points in X. If X is not instantiated as described above, an error message is displayed and the call fails.

findall

 

The set of X's such that P is provable is S.

 

findall(X, P, S)

 

Multiple solutions of a problem can be collected by repeatedly backtracking and gradually building up a full list of the solutions. The findall builtin predicate is provided to automate this process. Formally findall means:

 

"S is the set of all instances of X such that P is provable, where that set is non-empty".

 

The term P specifies a goal or goals as in call(P). S is a set of terms represented as a list of those terms, without duplicates, in the standard order for terms. The variables appearing in the term X should not appear anywhere else in the clause, except within the term P. For example, the call:

 

findall(X, mode(X), S).

 

may produce this solution:

 

S = [pre_flight, takeoff, cruise]

 

If there are uninstantiated variables in P that do not also appear in X, then a call to this builtin predicate may backtrack, generating alternate values for S. These values correspond to different instantiations of the free variables of P.

retract

 

The first clause in the program that matches C is erased.

 

retract(C)

 

C must be initially instantiated to a non-variable symbol. The predicate may be used in a non-determinate fashion, i.e. it will successively retract clauses matching the argument through backtracking. The space occupied by retracted or abolished clauses will be recovered during garbage collection.

trace

 

Switch on debugging and start tracing

 

trace

 

The builtin trace turns on the debug mode, and the interpreter starts tracing from the next call to a user goal. If trace was given in a command on its own, the goal(s) traced will be those of the next commands. Since this is a one-time decision, a call to trace is necessary whenever tracing is required right from the start of an execution.
